# NewsAgent 测试清单与简报质量评估表

版本基线：2026-06-27  
适用范围：当前本地版 NewsAgent（采集、存储、排序、规则简报、LLM 简报、翻译、问答、反馈、邮件和 Windows 定时任务）

## 1. 使用方法

每次准备修改程序、增加信息源、更换模型或调整提示词时：

1. 先执行“发布阻断项（P0）”。
2. 再抽查“功能测试”和“简报质量评分”。
3. 分别评价规则版 `latest_rules.md` 与 LLM 版 `latest_llm.md`。
4. 将问题记录到文末的“问题与改进记录”。
5. 只有 P0 全部通过，且简报质量总分达到 80 分，才建议作为日常正式简报使用。

状态填写方式：

- `[ ]` 未测试
- `[x]` 通过
- `[!]` 失败或需要改进
- `[-]` 本轮不适用

严重度：

- `P0`：影响可信度、安全性或核心可用性，必须优先修复。
- `P1`：明显影响内容质量或稳定性，应进入近期修改。
- `P2`：体验优化或新功能候选。

## 3. 发布阻断项（P0）

以下任意一项失败，当日简报应标记为“不可信/不可正式投递”。

### 3.1 安全与配置

- [ ] `SEC-01` 配置文件、日志、数据库和 Git 变更中不存在 SMTP 密码、API Token、Cookie 等明文秘密。
- [ ] `SEC-02` SMTP 密码只从环境变量读取，缺失时程序给出清晰错误且不发送。
- [ ] `SEC-03` 日志不打印密码、完整认证头或私人邮件正文之外的敏感信息。
- [ ] `SEC-04` `doctor` 只报告“是否配置”，不显示秘密值。

### 3.2 编码与可读性

- [ ] `TXT-01` 中文标题、栏目、摘要中没有 `譛`、`縺`、`蟶`、`窶`、`�` 等乱码特征。
- [ ] `TXT-02` 日文新闻保持正常日文，或被完整翻译为目标语言，不出现 mojibake。
- [ ] `TXT-03` Markdown、终端、数据库和邮件中的同一文本显示一致。
- [ ] `TXT-04` URL 以 `https://` 或 `http://` 正常显示，不出现前缀损坏。
- [ ] `TXT-05` 中英日混排时，股票代码、机构名、产品名不被错误翻译。

建议自动检查：

```powershell
Select-String -Path data\outbox\latest*.md -Pattern '譛|縺|蟶|窶|�'
```

预期：没有匹配结果。若来源正文自身含这些字符，应人工确认，而不是直接放行。

### 3.3 事实、来源与时间

- [ ] `FACT-01` 每条事实性新闻都有可打开的来源 URL。
- [ ] `FACT-02` 简报中的数字、日期、涨跌幅、机构名称与来源一致。
- [ ] `FACT-03` LLM 没有加入 evidence 中不存在的原因、结论、预测或人物表态。
- [ ] `FACT-04` 新闻发布时间在设定的 48 小时窗口内；缺失时间的项目被明确标记。
- [ ] `FACT-05` 行情明确区分实时价、盘前、盘后和上一交易日收盘价。
- [ ] `FACT-06` 简报生成时间、最新采集时间和行情时间均带 `Asia/Tokyo` 时区。
- [ ] `FACT-07` Story ID 与 URL 在翻译前后完全保留且一一对应。
- [ ] `FACT-08` 医疗内容不把论文观察结果写成已确立的临床结论。

### 3.4 栏目完整性

- [ ] `COV-01` 市场栏目包含全球指数、美国行业、商品与外汇；无数据时明确说明。

- [ ] `COV-06` 实际入选 story 数达到配置要求；不足时在简报和日志中报警。

## 5. 简报内容质量评分

rules 和 llm 两版分别评分。每项按 1–5 分：

- 1：严重不可用
- 2：较差，需要大量人工修订
- 3：基本可用，但问题明显
- 4：良好，仅需小幅修订
- 5：可直接使用

| 维度 | 权重 | 评分要点 |
|---|---:|---|
| 事实准确性 | 20 | 数字、日期、人物、事件与来源一致，无幻觉 |
| 时效性 | 12 | 新闻和行情足够新，时间状态解释准确 |
| 来源可信与可追溯 | 12 | 每条有可靠来源，链接可打开，官方/一手来源优先 |
| 覆盖完整性 | 12 | 市场、地区新闻、医疗、AI 均达到预期 |
| 重要性与相关性 | 12 | 真正重要的信息排在前面，符合用户关注方向 |
| 去重与信息密度 | 8 | 无重复事件、重复 URL 和低价值填充 |
| 摘要质量 | 8 | 说明“发生了什么、为什么重要”，不过度推断 |
| 语言与可读性 | 8 | 无乱码，结构清楚，翻译自然，长度适中 |
| 行动/跟踪价值 | 5 | 能指出值得继续关注的变量或后续事件 |
| 一致性与稳定性 | 3 | 多次运行结构稳定，rules/llm 差异可解释 |

计算方式：

`加权总分 = Σ（单项评分 ÷ 5 × 权重）`

判定：

- 90–100：可直接作为正式简报。
- 80–89：可用，建议少量人工抽查。
- 70–79：仅供内部参考，应安排改进。
- 60–69：质量不稳定，不建议自动投递。
- 60 以下：不可用，应先修复 P0 问题。

## 7. 新信息源验收

每增加一个 source，至少完成：

- [ ] `SRC-01` URL 可访问，许可和使用方式合理。
- [ ] `SRC-02` 连续三次采集成功。
- [ ] `SRC-03` 标题、摘要、发布时间、链接和语言解析正确。
- [ ] `SRC-04` 内容不是长期旧闻、广告、导航页或纯视频标题垃圾。
- [ ] `SRC-05` source 的 category、region、tier、priority 和 tags 正确。
- [ ] `SRC-06` 与现有来源重复度可接受。
- [ ] `SRC-07` 失败不会影响其他来源。
- [ ] `SRC-08` 有健康检查、最近成功时间和失败次数。
- [ ] `SRC-09` 能说明该来源补足了哪个信息缺口。

## 8. 回归与异常场景

- [ ] `ERR-01` 无网络。
- [ ] `ERR-02` 单个 RSS 返回 404、500、HTML 或损坏 XML。
- [ ] `ERR-03` API 限流 429。
- [ ] `ERR-04` SSL 证书错误。
- [ ] `ERR-05` Ollama 未启动。
- [ ] `ERR-06` 模型不存在或加载超时。
- [ ] `ERR-07` 模型返回空文本、超长文本或非法 Markdown。
- [ ] `ERR-08` 数据库锁定、损坏或磁盘空间不足。
- [ ] `ERR-09` SMTP 登录失败、超时或收件人拒收。
- [ ] `ERR-10` 一轮采集没有任何新内容。
- [ ] `ERR-11` 只有市场数据，没有新闻。
- [ ] `ERR-12` 新闻数量少于 min_stories。
- [ ] `ERR-13` 来源时间在未来或时区缺失。
- [ ] `ERR-14` 标题为空、URL 为空、摘要包含恶意 HTML。
- [ ] `ERR-15` 新闻正文或标题包含提示注入文本时，LLM 仍只把它当数据。

## 9. 自动化测试补充优先级

### 第一批：必须尽快增加

1. 编码/乱码检测测试。
2. 同 URL 和相似标题去重测试。
3. story 多来源合并测试。
4. LLM 输出栏目完整性测试。
5. LLM 每条引用与 story ID 校验测试。
6. prompt evidence 截断不遗漏后部栏目测试。
7. 邮件秘密不进入配置/Git 的扫描。
8. market 数值与 quote timestamp 一致性测试。
9. `min_stories` 实际生效测试。
10. `delivery_logs` 成功/失败记录测试。

### 第二批：稳定性

1. 各 collector 的 mocked HTTP 测试。
2. 超时、429、损坏 XML 和空响应测试。
3. daily 原子写文件测试。
4. feedback 排序影响及衰减测试。
5. 翻译分块边界、Markdown 和数字保护测试。
6. 计划任务脚本退出码与日志测试。

### 第三批：产品质量

1. 基于人工标注样本的排序准确率。
2. 简报重复率、引用覆盖率和栏目覆盖率自动统计。
3. rules 与不同模型的固定 benchmark。
4. 来源健康度和自动降级。
5. 历史简报趋势和用户反馈效果评估。

## 10. 测试结果如何指导修改

| 测试结果 | 优先修改方向 |
|---|---|
| 乱码或混合编码 | 先修复源文件/数据库/终端/邮件 UTF-8 链路，再讨论提示词 |
| LLM 栏目遗漏 | 按栏目组织 evidence、分段生成、最后合并，不直接截断 JSON |
| LLM 幻觉或无引用 | 改为结构化 JSON 输出；逐条验证 story ID、URL 和数字 |
| 重复率高 | URL canonicalization、标题相似度、事件聚类、多来源合并 |
| 重要新闻排位低 | 调整 source tier、category weight、关键词和用户反馈权重 |
| 某地区长期缺失 | 增加/修复该地区来源，并建立 source health dashboard |
| 医疗摘要不可靠 | 优先官方/论文来源；增加研究类型、样本量和证据等级字段 |
| 行情时间经常误解 | 在数据模型中明确 price_type、market_state、quote_time |
| 问答相关性差 | 改用 FTS/向量检索；禁止无结果时返回完全无关内容 |
| 邮件偶发重复 | 增加 delivery idempotency key、状态日志和有限重试 |
| 每日任务静默失败 | 增加失败通知、连续失败计数和源健康报告 |
| 人工仍需大量浏览 | 增加“为什么重要”“后续关注”“相比昨日变化”栏目 |

## 11. 新功能立项判断

只有当对应基础测试稳定后再开发：

| 新功能 | 启动条件 |
|---|---|
| Breaking Alert | 时间、去重、重要性评分和投递幂等已稳定 |
| Deep Dive | 引用校验、问答相关性和长文本证据组织已稳定 |
| HTML/网页简报 | Markdown 内容质量和编码先通过 |
| 趋势图表 | 行情历史存储、时间戳和单位准确 |
| 多渠道投递 | 邮件日志、重试和秘密管理先完成 |
| 自动个性化 | 反馈可撤销、可解释，并有权重衰减 |
| 云端运行 | 秘密管理、健康监控、备份和成本限制完成 |
| 向量搜索 | 先建立固定问答评测集，证明现有检索确有不足 |
